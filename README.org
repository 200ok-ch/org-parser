** What does this project do?

=org-parser= is a parser for the [[https://orgmode.org/][Org mode]] markup language for Emacs.

It can be used from JavaScript, Java, Clojure and ClojureScript!

** Why is this project useful / Rationale

Org mode in Emacs is implemented in [[http://git.savannah.gnu.org/cgit/emacs/org-mode.git/tree/lisp/org-element.el][org-element.el]] ([[https://orgmode.org/worg/dev/org-element-api.html][API
documentation]]). The [[https://orgmode.org/worg/dev/org-syntax.html][spec for the Org syntax is written in prose]].

This is already great work, yet it has some drawbacks:

1. The spec is not machine readable. Hence, there can be drift between
   documentation and implementation. In fact, during the development
   of [[https://github.com/200ok-ch/organice/][organice]], our web-based Org implementation with great mobile
   phone support, and =org-parser= we have encountered drift.
2. org-element.el is naturally written in Emacs lisp and makes strong
   use of Emacs as a text-processor. Hence, its code can only be used
   within Emacs.

While writing the official spec already is an amazing effort in the
standardization of the Org format, the power of Org is so enticing
that many want to use it outside of Emacs, as well. Since
org-element.el only runs in Emacs, this caused a myriad of
implementations for other platforms (JavaScript, Rust, Go, Java, etc)
to have been created. Most implementations are only partial, and more
importantly each of them creates another island. Since they are just
as programming language dependent as org-element.el, it is impossible
to share logic between them.

=org-parser= aims at alleviating both these issues. It specifies the
syntax in ANTLR grammars, which remain close to EBNF-style notation
while giving us a mature parser toolchain, robust error handling, and
multi-runtime generation for JVM and JavaScript. The reference
implementation is done in a way that it runs on the established
virtual machines of Java and JavaScript. Hence,
=org-parser= can be used from all programming languages running on
those virtual machines. =org-parser= provides a higher-level data
structure that is easy to consume for an application working with Org
mode data. Even if your application is not running on the Java or
JavaScript virtual machines, you can embed =org-parser= as a
command-line application. Lastly, =org-parser= brings a strong test
suite to document the reference implementation in yet another
unambiguous way.

It is our aim that =org-parser= can be the foundation on which many
Org mode applications in many different languages can be built. The
applications using =org-parser= can then focus on implementing user
facing features and don’t have to worry about the implementation of
the Org syntax itself.

** Architecture

The code base of =org-parser= is split into four namespaces:

- org-parser.core (top level API, i.e. =read-str=, =write-str=)
- org-parser.parser (parser API and parser contract wrappers)
- org-parser.transform (transforms parser AST into a higher-level structure)
- org-parser.render (serializer/writer)

ANTLR currently parses document and line-level structure. Several
inline constructs (for example parts of text, links, timestamps, and
table normalization) are still handled by direct parser helpers after
ANTLR returns a structural AST. This split is intentional for now and
is documented further in [[file:doc/parser-architecture.md][doc/parser-architecture.md]].

Thus =org-parser= has become a misnomer in the sense, that it now
strives to be =clojure/data.org= (after the pattern of existing Clojure
libraries like =data.json=, =data.xml=, =data.csv=, etc) providing
reader as well as writer capabilities for the serialization format
=org=.

* Project State

This project is work-in-progress. It is not ready for production yet
because the structure of the AST (parse tree) can still change.

The biggest milestones are:

- [X] Finish ANTLR grammar/parser coverage for most Org mode syntax
      - [X] Headlines
      - [X] Org mode =#+*= stuff
      - [X] Timestamps
      - [X] Links
      - [X] Text links
      - [X] Footnotes
      - [X] Styled text
      - [X] Drawers and =#+BEGIN_xxx= blocks
      - [ ] Nested markup (see #12)
- [X] Setup basic transformation from the parse tree to a higher-level structure.
- [-] Transformations to higher-level structure: catch up with
      features that are already supported by the parser.
- [-] Render parsed org file with =write-str=

*It can already be useful* for you:
E.g. if your script needs to parse parts of Org mode features, our
ANTLR grammar/parser probably already supports that. The grammar stays
EBNF-like and readable, while ANTLR gives us strong tooling across CLJ
and CLJS runtimes. Do not underestimate
e.g. timestamps. Use our well-tested parser to disassemble it in its
parts, instead of trying to write a poor and ugly regex that is only
capable of a subset of Org mode's timestamps ;)

Don't hesitate to contribute!

* Development

=org-parser= uses ANTLR-generated parsers as the default backend on
both CLJ and CLJS. This keeps grammar behavior aligned across JVM and
JavaScript runtimes while preserving the existing public API and test
suite expectations.

** Large-document parsing strategy

For most inputs, =org-parser= uses the ANTLR backend.

For very large full-document parses, =org-parser= uses a fast Clojure
path in the public parser API:

- Trigger conditions: default start rule =:S= and input size >=
  =1,000,000= bytes
- Implementation: =src/org_parser/parser.cljc=
  (=parse-large-doc-fast=)
- Metadata marker: =(:backend-used (meta result))= is =:fast-large=

Rationale:

- Full ANTLR tree construction on multi-megabyte files has much higher
  latency and memory overhead.
- The fast path preserves the top-level =:S= shape and keeps parsing
  responsive for large inputs, while ANTLR remains the default backend
  for detailed grammar-driven parsing.

Tradeoff:

- The fast path is intentionally coarse: it builds headline, content,
  and empty-line nodes from lines rather than running full ANTLR rule
  traversal for every inline construct.
- If you need strict grammar behavior for a specific rule, parse that
  rule explicitly via =:start= so it goes through ANTLR.

** Benchmark workflow

Use the benchmark namespace directly from Clojure:

#+BEGIN_SRC shell
# warmup=5 measured=20
lein run -m org-parser.benchmark 5 20
#+END_SRC

Run large-file regression tests together with benchmarking:

#+BEGIN_SRC shell
lein test :only org-parser.large-file-test/parse-large-unique-readme-derived-document
lein test :only org-parser.large-file-test/parser-accepts-large-unique-readme-derived-document
#+END_SRC

For maintainers: parser layering, runtime adapters, normalization,
and transform boundaries are documented in
[[file:doc/parser-architecture.md][doc/parser-architecture.md]].
Grammar scope and future migration candidates are tracked in
[[file:doc/grammar-roadmap.md][doc/grammar-roadmap.md]].

** Prerequisites

Please install [[https://clojure.org/guides/getting_started][Clojure]], [[https://leiningen.org/][Leiningen]], and Node.js (with npm).

Leiningen will pull JVM dependencies. For CLJS/node tests, install
JavaScript dependencies with =npm install=.

** Testing

Running the tests:

#+BEGIN_SRC shell
# Clojure
lein test
# CLJS (single run)
npm install
./script/gen-antlr-js.sh
lein doo node once
#+END_SRC

Parser API quick checks:

#+BEGIN_SRC clojure
(require '[org-parser.parser :as parser])

;; discover supported parser start rules
(parser/supported-start-rules)

;; parse an entire document (default start rule :S)
(parser/parse "* headline\nbody")

;; parse specific start rule
(parser/parse "#+TODO: TODO | DONE" :start :todo-line)
#+END_SRC

Generate ANTLR parser artifacts (required for ANTLR backend work):

#+BEGIN_SRC shell
./script/gen-antlr.sh
./script/gen-antlr-js.sh
#+END_SRC

** Contributor Workflow

For parser/grammar changes, use this sequence:

#+BEGIN_SRC shell
# 1) Install dependencies
lein deps
npm install

# 2) Regenerate parser artifacts
./script/gen-antlr.sh
./script/gen-antlr-js.sh

# 3) Run static checks and tests
clj-kondo --lint src test --parallel
lein test
lein doo node once
#+END_SRC

Common troubleshooting:

- If CLJ tests fail with missing generated classes, run
  =./script/gen-antlr.sh=.
- If CLJS tests fail with missing JS parser runtime modules, run
  =npm install= and =./script/gen-antlr-js.sh=.
- If parser behavior differs between runtimes, run both test commands
  above and check parser parity tests first.

* Usage
  :PROPERTIES:
  :CUSTOM_ID: usage
  :END:

At the moment, you can run =org-parser= from Clojure, ClojureScript,
or Java. Other targets which are hosted on the JVM or on JavaScript
are possible.

** Clojure Library

 #+BEGIN_SRC clojure :exports both
   (ns hello-world.core
     (:require [org-parser.parser :refer [parse]]
               [org-parser.core :refer [read-str write-str]]))

   (prn (parse "* Headline"))
   (prn (read-str "* Headline"))
   (println (write-str (read-str "* Headline")))
 #+END_SRC
